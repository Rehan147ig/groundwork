Phase A · Bank-Stack Selection

Where Groundwork tests best against a bank-shaped Java backend.

A working comparison of the four banking systems most often proposed as test targets — plus the synthetic baseline Groundwork already supports — scored on what actually matters for proving runtime AI access-control.

What Groundwork is

The AI runtime access-control layer. It sits between an AI agent and the document stores the agent retrieves from. Every chunk is checked against the company’s real permission graph at query time, fail-closed, with a tamper-evident audit trace.

What Groundwork is not

Not a core banking system. It does not hold accounts, balances, or ledgers. It protects the AI’s document retrieval surface — policies, memos, customer files, KYC packets — over whatever knowledge stores the bank already runs.

01 · Integration Architecture

Push vs. Pull — how Groundwork plugs into a bank’s permission graph.

Groundwork’s job at integration time is to learn who can see what from the bank’s systems of record, then keep that view current as people join, change roles, and leave. There is no single right way to do that — banks run a mix of open-API platforms, modern SaaS, and proprietary cores that don’t expose a clean permission API at all. Groundwork supports three patterns, deliberately, so the integration choice fits the source system rather than the other way around.

Pattern A

API Pull Connectors

Bank System — GET /users, /permissions → Groundwork poll on schedule + delta tokens

Groundwork periodically reads users, groups, and document grants from the source via its REST API, computes a diff against the OpenFGA tuples it already holds, and writes only the changes. Source of truth stays in the bank’s system; Groundwork is a derived view.

Fits Apache Fineract · Open Bank Project · Microsoft Graph · Okta · any system with documented read APIs and stable schemas.
Pattern B

Hybrid — API + Webhook

Bank SaaS — GET (baseline) → Groundwork + POST /sync/event on every change

One pull builds the baseline, then the source pushes change events as they happen. Near-real-time without polling pressure, with the API as a fallback when an event is lost. The right pattern when the source is modern SaaS and revocation latency matters.

Fits Mambu · modern SaaS cores · HRIS systems with event streams · any source where seconds, not minutes, matter for revocation.
Pattern C

Webhook Push API · Sidecar Ingest

Proprietary Core — POST /sync/event → Groundwork bank-side sidecar emits ACL events

When the source has no readable permission API — or its API is too closed, slow, or licensed-per-call to query — the bank installs a thin sidecar that watches its core’s change feed and posts events to Groundwork’s ingest endpoint. Push-only, Groundwork never reaches into the core. Critical for highly customized environments and regulators who require nothing pulls from the core.

Fits Temenos T24 / Transact · FIS / Fiserv mainframe cores · any heavily customized or air-gapped system where pull is disallowed.

03 · System-by-System

The honest read on each option.

Per-system context for the bank stakeholders who will dig into why a particular target was chosen or set aside. The order is the matrix order; the takeaway in each card is the position Groundwork takes externally.

Apache Fineract

Open-source core banking · Java 21 / Spring Boot
Useful later
Strength
The cleanest real Java banking engine to point at when banks ask whether Groundwork integrates with a Java stack. Spring Boot, plugin JARs, well-documented REST API.
Weakness
It’s a ledger, not a knowledge store. It tracks accounts, loans, transactions — not the policies and memos Groundwork is built to gate. Adding it now buys integration credibility, not test coverage of the actual claim.
Pattern
Pattern A — API Pull Connector. Pull users and customer-record visibility through Fineract’s REST API into OpenFGA tuples.
External read: “Groundwork has an Apache Fineract pull connector for banks running open-source cores.” A milestone for later, not the first demo.

Temenos

Commercial Tier-1 core · Java + Groovy extensibility
Future commercial
Strength
The platform name on a deck that opens enterprise banking doors. Real Java extensibility surface. The system most large banks actually run for the things Groundwork would protect.
Weakness
No realistic free sandbox path. Without a Temenos partnership, you cannot demo against it. Treat it as a commercial integration target once Groundwork has paying customers running Temenos.
Pattern
Pattern C — Webhook Push / sidecar. Temenos environments are too closed for pull; a bank-side sidecar that emits ACL change events to Groundwork is the only realistic shape.
External read: “Groundwork integrates with Temenos via a push-based sidecar pattern for banks that cannot expose pull APIs to the AI layer.” A pitch, not a demo — yet.

Mambu

Cloud-native SaaS core · API-first
Showcase later
Strength
The cleanest modern SaaS banking architecture and the closest to what newer banks and fintechs actually run. Strong API surface, real event/webhook story.
Weakness
Sandbox access requires partner provisioning, and Mambu still doesn’t produce the document surface Groundwork protects. The integration story is strong; the test coverage of the claim is thin.
Pattern
Pattern B — Hybrid. Pull a baseline through Mambu’s API, then subscribe to webhooks for change events. Near-real-time revocation without polling.
External read: “For Mambu-based banks Groundwork uses a hybrid API + webhook pattern, baseline pull plus event-driven updates.” A second-wave demo target.

Open Bank Project

Open-source banking API sandbox · Scala/Java
Optional
Strength
The best free sandbox for OAuth2 / consent / mTLS / JWS flows — the developer-facing edge of a bank, rather than its core. Useful when the conversation is about consent rather than retrieval.
Weakness
Like Fineract it’s API-and-consent shaped, not document shaped. The retrieval claim Groundwork makes does not land naturally against OBP without a synthetic document layer on top.
Pattern
Pattern A — API Pull. Pull consents and account-access scopes from OBP into OpenFGA.
External read: A useful side-demo for the open-banking / consent conversation. Not the headline demo.
Recommended for the next demo

Synthetic bank-document corpus · OpenFGA permission graph · Java Spring Boot client

Build now
What it is
A curated, clearly-synthetic library of bank-shaped documents — credit memos, KYC packets, internal policy memos, audit findings, customer correspondence — loaded into Qdrant; a realistic role and group model loaded into OpenFGA; and a Java Spring Boot application that plays the part of an internal bank tool calling Groundwork over its REST API and MCP endpoint.
Why it wins
It tests the actual claim. Fail-closed only matters if there is a document to fail closed on. Shadow mode only matters if there is a believable role boundary. Audit only matters if there is a sensitive query to audit. None of the four real systems gives Groundwork all three without a synthetic document layer added on top — so the synthetic layer is the work, regardless of which core sits behind it.
Reuse
Groundwork’s existing loadtest -mode=seed tool already seeds a bank-shaped users/docs/OpenFGA dataset. The build is curating that into deck-grade content, adding the Java client, and writing a tight demo script — not standing up a new core.
Pattern
Pattern A — direct OpenFGA seed. In production the same tuples come from a Fineract or Microsoft Graph pull connector or a Temenos push sidecar; for the demo the source is the seed tool, which keeps the demo deterministic and reproducible on a laptop.
External read: “Here is a realistic bank knowledge surface with realistic role separation. Watch what Groundwork lets each persona retrieve, what it blocks, what shadow mode would have blocked, and the tamper-evident audit trail it leaves behind.”
04 · Recommendation

Build the synthetic baseline now. Earn the rest later.

Phase B · Decision

Synthetic banking corpus + Java Spring Boot client — behind Groundwork’s existing runtime.

The four real systems each answer a single question well: Fineract answers do you do Java cores; Temenos answers do you sell to Tier 1; Mambu answers do you do modern SaaS; Open Bank Project answers do you do consent. None of them, on their own, answers the question Groundwork actually has to answer for a bank security buyer:

  • Can you stop the AI from retrieving a document this user is not allowed to see?
  • Can you run silently first and show me what would have leaked under my current setup?
  • Can you give me an audit trail that survives my regulator?

The synthetic baseline answers all three, on a laptop, in days, against the real Groundwork runtime — with the Java Spring Boot client carrying the “and yes, this works from your Java stack” story for any bank that asks. Then, once the demo lands, the Fineract pull connector becomes the next logical milestone, followed by the Temenos sidecar pattern, then Mambu, in that order, driven by which pilot conversation actually progresses.

What this demo proves

Fail-closed against a real document set, shadow mode with believable false-positive numbers, and an audit chain a bank auditor can hash-verify themselves.

What this demo carefully avoids

Pretending to be integrated with a bank we are not integrated with. The corpus is clearly synthetic. Every screen says so.

What it sets up next

The Fineract pull connector and the Temenos push sidecar slot in behind the same OpenFGA seed surface — no rewrite, just a different source.
